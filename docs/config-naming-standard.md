# Unified Configuration Naming Standard

## Overview

This document defines a single, consistent naming convention for configuration keys across all layers of the application—from .NET code to Kubernetes deployments. This standard eliminates confusion and enables automatic configuration mapping through the .NET configuration system.

## Convention Table

| Layer | Separator | Format | Example |
|-------|-----------|--------|---------|
| .NET code | `:` | `Section:Key` | `CosmosDb:Endpoint` |
| appsettings.json | nested JSON | `{ "Section": { "Key": "" } }` | `{ "CosmosDb": { "Endpoint": "" } }` |
| Env vars (Helm/K8s) | `__` | `Section__Key` | `CosmosDb__Endpoint` |
| Key Vault secrets | `--` | `Section--Key` | `CosmosDb--Endpoint` |
| config-sync | automatic | KV `--` → nested JSON | `CosmosDb--Endpoint` → `{ "CosmosDb": { "Endpoint": "" } }` |

## Why Double Separators?

The .NET configuration system has built-in support for automatic hierarchical mapping:
- Environment variables use `__` (double underscore) to map to `:` hierarchy in code
- Key Vault secrets use `--` (double hyphen) for the same purpose

This automatic mapping eliminates the need for manual parsing code in every service. Without these conventions, each component would require custom configuration binding logic, leading to inconsistency and maintenance burden.

**Example**: The KV secret `CosmosDb--Endpoint` automatically maps to `config["CosmosDb:Endpoint"]` in code, and to `{ "CosmosDb": { "Endpoint": "" } }` in JSON.

## Per-Component Configuration

Each service's configuration files are gitignored and generated at runtime:

- **`appsettings.Development.json`** — Generated by config-sync with real Azure endpoints for local development. Never committed to source control.
- **`appsettings.json`** — Generated at deployment time (Helm/K8s) with production configuration. Never committed to source control.

All `appsettings*.json` files are listed in `.gitignore`. Developers run config-sync to populate their local Development config from Key Vault. Each component's README documents its required configuration keys.

This approach ensures:
- No configuration files are checked into source control
- Developers run one command (`config-sync`) to set up local config
- Production deployments use real configuration from Key Vault
- Configuration is immutable once deployed (generated, not modified)

## Adding a New Configuration Key

When adding a new configuration key to the system, follow this checklist:

1. **Define in Key Vault** — Add the secret to Terraform with the naming convention `Section--Key` (e.g., `CosmosDb--ConnectionString`)
2. **Update config-sync** — Add the key to your component's manifest so config-sync picks it up during deployment
3. **Update component README** — Document the new key in the component's Configuration table
4. **Reference in code** — Read the value using `config["Section:Key"]` in your .NET application
5. **Update Helm values** — Add the environment variable mapping to `values.yaml` as `Section__Key` for Helm chart templating

This checklist ensures consistency across all configuration sources and prevents deployment failures due to missing configuration.
